# ร่างสไลด์นำเสนอลูกค้า — โครงการ PM Application (ชุดเดียว สั้น ชัด)

เอกสารนี้เป็น **สคริปต์ bullet ทีละสไลด์** สำหรับคัดลอกไป PowerPoint / Google Slides  
**ชุดเดียว** รวมเรื่องธุรกิจ + โครงสร้างเบื้องหลังให้ลูกค้าเห็นภาพรวมในครั้งเดียว — เป้า **ประมาณ 10 สไลด์ / 10–15 นาที** (พูดไม่รีบ ไม่ต้องอ่านทุก bullet)

**อ้างอิงใน repo:** [`PROJECT_PLAN.md`](PROJECT_PLAN.md) · [`INSTALL_SOP_TAILSCALE_DOCKER.md`](INSTALL_SOP_TAILSCALE_DOCKER.md) · [`PROGRAM_FLOW.md`](PROGRAM_FLOW.md)

**ชื่อเทคบนสไลด์ (สอด repo):** เว็บ **React + TypeScript (Vite)** · **Node.js (API)** · **MariaDB พอร์ต 3307** — ไม่ใช้คำว่า Next.js เว้นแต่ทีมเปลี่ยนจริง

**Telegram / แจ้งเตือน:** ถ้ายังไม่ล็อกใน SRS ให้ไม่ใส่บนสไลด์หลัก หรือใส่บรรทัดเดียวว่า “phase ถัดไปตามตกลง”

## สไลด์ 1 — ปก

- **ระบบ PM Application — งานซ่อมบำรุง (Plant Maintenance)**
- ลูกค้า: **บริษัท เป๊ปซี่โคล่า (ไทย) เทรดดิ้ง จำกัด (ลำพูน)**
- ผู้นำเสนอ: **บริษัท เอส.วาย. อินเตอร์แอคทีฟ ดีเวลลอปเมนท์ จำกัด**
- วันที่: *(กรอก)*

---

## สไลด์ 2 — วันนี้สรุปอะไรบ้าง (สามข้อ)

- **ทำไมต้องปรับ** และภาพระบบใหม่เป็นอย่างไร *(สไลด์ถัดไป — ใช้เวลาไม่กี่นาที)*
- **ลูกค้าได้ใช้อะไร** และ **นำเข้าไฟล์ SAP** ทำงานยังไงแบบเข้าใจง่าย
- **แผนส่งมอบ** + **สิ่งที่ขอจากลูกค้า** + ช่องถามตอบ

---

## สไลด์ 3 — จากเดิม สู่ทิศทางใหม่ (สองคอลัมน์ในหัวใจเดียว)

**เดิม (สรุป)**

- ข้อมูลกระจายใน **ไฟล์และระบบเก่า** — ติดตามและต่อยอดยาก
- ตรวจสอบย้อนหลังหรือปรับปรุงโดยไม่กระทบงานเดิม **ทำได้ยาก**

**ใหม่ (สรุป)**

- **เว็บแอป** ที่ใช้งานผ่านเบราว์เซอร์ + **ชั้นกลาง (API)** คุมสิทธิ์และกฎ
- **ฐานข้อมูลโครงการแยก** จากระบบเดิม — เก็บประวัติและนำเข้าได้เป็นระบบ

---

## สไลด์ 4 — ภาพรวม “หนึ่งแผนภูมิ” (ธุรกิจ + โครงสร้างในสไลด์เดียว)

*แนะนำวาดเป็นแนวตั้งหรือแนวนอน: กล่อง 4–5 กล่อง ลูกศรเดียว — ข้อความด้านล่างเป็นคำอธิบายสั้น*

- **ผู้ใช้ (โรงงาน / สำนักงาน)** → เปิด **เว็บแอป** (พอร์ต **3000**)
- เว็บคุยกับ **API** (พอร์ต **5000**) — ตรวจสอบสิทธิ์ก่อนบันทึกข้อมูล
- API เก็บลง **MariaDB โครงการ** (พอร์ต **3307** — **ไม่ไปชน** พอร์ตฐานข้อมูลเดิม เช่น 3306)
- **ทั้งหมดรันบนเครื่องลูกค้า** โดยใช้ **Docker** — แยกจากโปรแกรมเก่าบน Windows ชัดเจน
- ข้อมูลโปรเจกต์เก็บบน **Drive D** (แยกจาก C) — มีโซนสำรองตามแผน ~350 GB *(ปรับตามเครื่องจริง)*
- **การเข้าถึงจากภายนอก / ทีมพัฒนา:** ใช้ **Tailscale** เป็นอุโมงค์เข้ารหัส — **ต้องสอดคล้องไฟร์วอลล์โรงงาน** (รายละเอียดใน SOP)

---

## สไลด์ 5 — ลูกค้าใช้งานอะไรได้ (เมนูหลัก ภาษางาน)

- **เข้าสู่ระบบ** ตามบทบาท
- **นำเข้าไฟล์จาก SAP** — ใบงาน (IW37N), Confirm WO, GI, GR *(ตาม scope ที่ล็อก)*
- **ดูใบงาน / ค้นหา / ปฏิทิน / มอบหมาย** + Reason ตาม requirement
- **ยืนยันใบงาน** และเห็นสถานะหลังประมวลผล
- **รายงาน / template SAP** และ **แดชบอร์ด KPI** *(ตาม scope)*
- **จัดการผู้ใช้และสิทธิ์** *(เมื่อเปิดในโครงการ)*

---

## สไลด์ 6 — นำเข้าไฟล์ SAP: สี่ขั้น (จำง่าย)

1. เลือก **ประเภทข้อมูล** + ไฟล์ที่ export จาก SAP  
2. **อัปโหลด**  
3. ระบบ **ตรวจและประมวลผล** — แสดงสำเร็จ / ผิดพลาด  
4. ดู **ประวัติการนำเข้า** และรายละเอียด error ได้

## สไลด์ 8 — ใครช่วยอะไร + ขอจากลูกค้า

**บทบาทสั้น ๆ**

- **ลูกค้า / โรงงาน:** ไฟล์ตัวอย่าง SAP, UAT, ยืนยันพฤติกรรมจริงหน้างาน  
- **IT ลูกค้า:** เครื่อง, ไฟร์วอลล์, สำรองข้อมูล, ติดตั้งตาม SOP  
- **PO / โครงการ:** ลำดับความสำคัญและเกณฑ์รับงาน  
- **ทีม S.Y.:** ออกแบบ พัฒนา ทดสอบ ส่งมอบ  

**ขอในช่วงนี้**

- ไฟล์ SAP **จริง** แต่ละประเภทที่ใช้ในโรงงาน  
- ยืนยัน **รุ่นเอกสาร requirement** ที่ล็อก scope  
- ชื่อ **ผู้ทำ UAT** + ช่วงเวลา  
- ข้อมูล **plant** และกฎที่ต้องคุยกับทีม SAP

## สไลด์ 9 — ความเสี่ยงที่ควรรู้ร่วมกัน (ไม่ยาว)

- รูปแบบไฟล์ SAP **ต้องตรงตัวอย่างที่ล็อก** — เปลี่ยนแบบไฟล์ต้องประเมินใหม่  
- ส่งข้อมูลกลับ SAP อัตโนมัติ **อาจแยก phase** ตามข้อตกลง MM/PM  
- ฟีเจอร์บางอย่าง **ขึ้นกับการล็อกรูปแบบไฟล์** กับ SAP  

---

## สไลด์ 10 — สรุป + Q&A

**สรุปสามบรรทัด**

- **ข้อมูลเข้าเป็นระบบ** — ใช้งานผ่านเว็บ — **ตรวจสอบย้อนหลังได้**  
- **โครงสร้างบนเครื่องลูกค้า** แยกชัด: D:, Docker, พอร์ต 3000 / 5000 / 3307  
- ความสำเร็จต้องมี **ไฟล์ตัวอย่าง + UAT** จากลูกค้า  

**ถามตอบ**

- เปิดคำถาม — สรุป action (ชื่อ + วันครบ) — นัดครั้งถัดไป
